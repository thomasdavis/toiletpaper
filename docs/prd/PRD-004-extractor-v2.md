# PRD-004 · Extractor v2 — Structured Claim Schema and Testability

| | |
|---|---|
| Status | Draft |
| Created | 2026-05-02 |
| Owner | toiletpaper engine |

## Problem

The current extractor (`@toiletpaper/extractor`, single LLM call over
text → `{ title, authors, abstract, claims }`) is producing low-fidelity
output that compromises the rest of the pipeline:

1. **Title smashed with subtitle.**
   `papers.title = "Tuṣāra, vaiśya are Yuezhi (Tocharian speakers) who
   migrated to Xinjiang in 4th m. BCE traded Maujavata soma; and
   returned to India ca. 30 CE"` — that's the full subtitle line. The
   first sentence of a paper is not its title.

2. **Authors empty.** `papers.authors = '{}'` for an authored paper.
   The extractor doesn't pull author metadata reliably.

3. **Title repeated as a claim.** `claims[0].text` equals the title
   verbatim with `confidence = 0.98`. The title row is not a claim.

4. **Confidence ≠ testability.** Claim 7 from the Tuṣāra paper —
   *"The persons engaged in Soma processing are named in ancient
   texts"* — gets `confidence = 0.93`. It is not falsifiable. The
   extractor does not separate "the LLM is confident the claim is
   stated" from "this claim can be operationalized as a test".

5. **No structured fields in Postgres.** The schema only has `text` and
   `confidence`. `predicate`, `value`, `unit`, `category`, `evidence`
   live solely in Donto, and Donto ingest is brittle (PRD-005), so on
   any Donto outage the structure vanishes from the system.

6. **Section/page provenance is missing.** A claim doesn't know which
   page or section it came from. The annotated view's fuzzy-match
   tries to find the claim text in the body; for paraphrased extracts,
   this fails silently.

## Goals

1. The extractor returns header metadata (title, authors, year, doi,
   arxiv id) and a typed list of claims with structured fields the
   rest of the pipeline can rely on.
2. Each claim carries an explicit `testability` score independent of
   `confidence`, and untestable claims are filtered out of the
   simulation queue automatically.
3. Each claim carries source provenance (page, section, span, model,
   model version, parser version, byte count) so the annotated view
   never has to fuzzy-match.
4. The relational `claims` table mirrors the structured fields so a
   Donto outage degrades only the bitemporal-graph features, never the
   core verdict pipeline.

## Non-goals

- A full PDF parser rewrite. We continue to use `pdf-parse` /
  `extractTextFromPdf` for text extraction. The improvements are at
  the LLM-extractor layer.
- OCR for scanned PDFs (deferred). Papers must have a text layer.
- Multilingual support beyond what the LLM gives us for free.

## Background / current state

`extractor/src/index.ts` exposes:
- `extractTextFromPdf(buffer) → { text, pageMap, ... }`
- `extractClaimsFromText(text, apiKey) → { title, authors, abstract, claims }`

`claims[i]` is `{ text: string, confidence: number, category?: string }`.
The route inserts only `text` and `confidence` into Postgres. `category`
is dropped on the floor in the relational store; only Donto sees it.

## Proposed design

### `ExtractionResult` schema

```ts
export interface ExtractionResult {
  paper: {
    title: string;             // single line, ≤ 240 chars
    authors: string[];         // surname, given-name? format
    year: number | null;
    venue: string | null;
    arxivId: string | null;
    doi: string | null;
    abstract: string | null;
    primaryDomain: PaperDomain;        // from PRD-001
    domainConfidence: number;
    bodyCharCount: number;
    pageCount: number;
  };
  claims: ExtractedClaim[];
  /** Tokens spent extracting; used by the LLM-budget accounting */
  tokensUsed: number;
  extractor: {
    model: string;             // e.g. "gpt-4o-mini-2026-04"
    extractorVersion: string;  // e.g. "0.4.0"
    parserVersion: string;     // pdf-parse version
  };
}

export interface ExtractedClaim {
  /** The exact sentence(s) lifted from the paper */
  text: string;
  /** Optional paraphrase that retains meaning, generated by the LLM */
  canonicalText: string | null;
  category: ClaimCategory;            // from PRD-001
  /** Probability the claim is faithfully extracted from the paper */
  confidence: number;
  /** Probability the claim can be operationalized as a test */
  testability: number;
  /**
   * Why the claim got that testability score; user-readable string.
   * E.g. "no quantitative content" or "scaling-law form".
   */
  testabilityReason: string;
  /** Structured form for quantitative claims */
  predicate: string | null;     // "muon-g-2-anomaly"
  value: string | null;         // "4.2"      (string to preserve precision)
  unit: string | null;          // "sigma"
  /** Free-form short evidence supporting the claim, lifted verbatim */
  evidence: string | null;
  /** Source provenance */
  span: {
    page: number;               // 1-indexed
    sectionPath: string[];      // ["3 Methods", "3.2 Setup"]
    charStart: number;          // offset into the extracted body text
    charEnd: number;
  };
}
```

### Extraction pipeline (4 stages)

The single-LLM-call extractor is replaced by a deterministic pipeline:

1. **PDF → structured text.** `pdf-parse` plus a custom layout
   classifier that emits `BodyBlock[]` with `{type, page, sectionPath,
   text}`. Sections are detected via heading regex + heuristic font-size.

2. **Header parser.** Pulls title, authors, abstract, year, venue,
   arxiv/doi from the first page using a small dedicated LLM prompt
   (gpt-4o-mini, JSON-schema output, max 800 input tokens). The first
   sentence of the abstract is not used as the title.

3. **Claim extractor.** Splits the body into windowed segments
   (≤ 6 000 chars, with 600-char overlap), and emits structured
   claims per window. Per-claim `confidence` is the LLM's stated
   confidence; `testability` is computed by a separate downstream
   classifier (next stage) so we don't conflate the two.

4. **Testability classifier.** A second pass runs over each claim with
   a focused prompt: *"Given the claim text and category, can this
   claim be tested against simulation, citation lookup, or a closed
   formula? Return testability ∈ [0,1] and a one-line reason."*
   Untestable claims (testability < 0.4 by default) are stored but
   **not** queued for simulation.

### `claims` table mirror

```sql
ALTER TABLE claims
  ADD COLUMN canonical_text text,
  ADD COLUMN category text NOT NULL DEFAULT 'unknown',
  ADD COLUMN testability double precision,
  ADD COLUMN testability_reason text,
  ADD COLUMN predicate text,
  ADD COLUMN value text,
  ADD COLUMN unit text,
  ADD COLUMN evidence text,
  ADD COLUMN page integer,
  ADD COLUMN section_path text[],
  ADD COLUMN char_start integer,
  ADD COLUMN char_end integer,
  ADD COLUMN extractor_model text,
  ADD COLUMN extractor_version text;

CREATE INDEX claims_paper_testable_idx
  ON claims (paper_id) WHERE testability >= 0.4;
```

The route writes both Postgres rows *and* (when Donto is reachable)
Donto quads. They no longer diverge.

### Untestable-claim handling

The orchestrator (PRD-001) filters its candidate set to
`claim.testability >= TESTABILITY_FLOOR` (env-tunable, default 0.4).
Claims below the floor are recorded in `simulations` once, with:

```
verdict: 'untested'
reason: claim.testabilityReason
method: 'router-untestable'
```

This means every claim has at least one row in `simulations`, and the
UI's per-claim "Why no verdict?" answer comes from the simulation
table, not from absence-of-row.

### Annotated view becomes deterministic

The annotated view stops fuzzy-matching the claim text into the body.
Instead it uses `(page, char_start, char_end)` from the claim row to
locate the exact span. The fuzzy fallback remains for legacy claims
without provenance, gated on a feature flag.

## Migration

1. Land the schema changes (additive — defaults are nullable / unknown).
2. Ship Extractor v2 behind `EXTRACTOR_VERSION=v1|v2` flag. Default
   `v1` until parity is checked.
3. New uploads run v2 by default; v1 is deprecated after 14 days.
4. Backfill: a `xtask reextract --paper=<id>` command re-runs v2 over
   the GCS-stored PDF and overwrites the extracted rows for that
   paper, preserving the paper's UUID. (See PRD-008 for re-runs.)

## Acceptance criteria

- For the Tuṣāra paper re-extracted with v2:
  - `papers.title` is one short line (≤ 60 chars in this case).
  - `papers.authors` is non-empty.
  - The title is not duplicated as `claims[0].text`.
  - Every claim row has `category`, `page`, `char_start/end` populated.
  - Claims like *"the persons engaged in Soma processing are named in
    ancient texts"* are stored with `testability < 0.4` and **not**
    routed to any simulator.
- A claim's annotated-view highlight uses
  `(page, char_start, char_end)` and matches exactly the source span;
  the fuzzy-match library is no longer in the bundle.
- A Donto outage during upload still produces fully populated
  `claims` rows in Postgres.
- `extractor.tokensUsed` and `extractor.model/version/parserVersion` are
  captured per claim (so we can compare claim quality across
  extractor versions).

## Phasing

## Telemetry

- `extractor.duration_seconds` histogram, labeled `version, model`
- `extractor.tokens_used` histogram
- `extractor.claims_per_paper` histogram
- `extractor.untestable_share` gauge (claims with testability<floor)
- A weekly report: number of papers re-extracted, claim-count diff,
  testability-share diff.

## Open questions

- Do we keep the canonical text in Postgres (storage cost) or generate
  on demand? Default: store; cheap.
- Should `evidence` be the full quoted paragraph or just the
  sentence? Sentence; full paragraph belongs in the annotated view.
- Multi-language papers — currently English-trained extractor. The
  Tuṣāra paper has Sanskrit + Tocharian transliterations interleaved
  with English; v2 must handle Unicode-heavy bodies cleanly.
